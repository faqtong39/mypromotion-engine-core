# 开源核心功能范围定义（mypromotion-engine-core）

> **版本**: v1.0  
> **日期**: 2026-05-20  
> **定位**: 纯 Python 促销计算引擎库，零框架依赖，`pip install` 即可用  
> **边界原则**: 开源核心 = "可计算"的能力；闭源 SaaS = "可运营"的平台

---

## 一、总体策略：提取什么、剔除什么

### 1.1 核心判断标准

| 判断维度 | 保留（开源核心） | 剔除（闭源 SaaS） |
|---------|---------------|-----------------|
| **是否纯数学/逻辑计算** | ✅ 保留 | ❌ 剔除 |
| **是否需要持久化存储** | ❌ 转为内存/dataclass | ✅ 留在 SaaS |
| **是否需要异步任务/消息队列** | ❌ 剔除 | ✅ 留在 SaaS |
| **是否是管理后台功能** | ❌ 剔除 | ✅ 留在 SaaS |
| **是否是多租户隔离逻辑** | ❌ 剔除 | ✅ 留在 SaaS |
| **是否是企业运营专属** | ❌ 剔除 | ✅ 留在 SaaS |

### 1.2 一句话总结

> 开源引擎只负责：**输入购物车 + 规则 → 输出应付金额 + 优惠明细 + 跳过原因**。  
> 所有需要数据库、Redis、Celery、Admin、Webhook 的功能，都留在闭源 SaaS。

## 二、保留的核心功能（8大模块）

### 2.1 Decimal 全链路精度 ✅

**保留理由**: 这是引擎最硬核的技术差异化，个人开发者用 float 凑活，企业级交易不能凑活。

**开源形态**:
- 所有金额字段使用 `Decimal`，禁止 float 参与计算
- `ROUND_HALF_UP` 统一量化到分（`0.01`）
- 阶梯价、百分比折扣、固定价等全部 Decimal 原生
- 提供 `apply_limits()` 工具方法：封顶、不超订单金额、不允许负折扣

**参考代码**: `discount_calculator.py` 中的 `DiscountCalculator.apply_limits()`

```python
from decimal import Decimal
from promotion_engine import Engine, CartItem

item = CartItem(sku="A001", price=Decimal("199.00"), qty=2)  # 禁止传 float
```

---

### 2.2 四层互斥检查 ✅

**保留理由**: 互斥是促销计算的灵魂，开源必须保留完整的互斥逻辑才能体现引擎价值。

**开源形态**（纯内存计算，去 Django ORM）:

| 层级 | 开源实现 | 说明 |
|-----|---------|------|
| L1 特殊互斥规则 | `SpecialMutexRule` dataclass + `MutexChecker._check_special_mutex()` | 规则A↔规则B的互斥/替换 |
| L2 强制叠加 | `stack_config.force_stackable` | 白名单机制 |
| L3 策略类型互斥组 | `MutexGroup` dataclass + 组内策略互斥 | 基础折扣互斥组、秒杀互斥组等 |
| L4 可叠加策略默认 | 空列表（默认全部参与互斥） | 由用户通过互斥组配置控制 |

**关键能力保留**:
- 双向互斥（先应用者保留）
- 单向互斥 + 替换回滚（高优先级替换低优先级，金额回滚重算）
- `stack_config` 白名单/黑名单（`stackable_with` / `mutex_with`）
- 互斥跳过原因透明化（每一条被跳过的规则都带原因）

**剔除**: 动态配置感知（`DynamicConfigManager` 的数据库轮询/缓存）→ 改为构造函数直接传入互斥组配置

---

### 2.3 计算顺序参数化 ✅

**保留理由**: 促销先 vs 券先 vs 自动最优，是电商核心场景，计算逻辑极轻量。

**开源形态**:
```python
engine = Engine(calculation_order="optimal")  # promotions-first / coupons-first / optimal
result = engine.calculate(cart, rules)
```

- `promotions-first`: 先算促销，再算券
- `coupons-first`: 先算券，再算促销
- `optimal`: 并行计算两种顺序，自动取应付金额最低的方案

**参考代码**: `engine.py` 中 `calculate()` 的三种模式实现

---

### 2.4 插件化架构 ✅

**保留理由**: 没有插件化就不是引擎，只是一个计算器。

**开源形态**（去 Django 导入，纯 Python 注册）:

```
promotion_engine/
├── plugins/
│   ├── base.py          # PluginBase / ConditionPlugin / ActionPlugin / ScopePlugin
│   ├── manager.py       # PluginManager（纯内存注册表）
│   └── builtins.py      # 15策略 × 12条件 × 12动作 × 6范围 的内置插件
```

**保留能力**:
- 条件插件基类 + 内置 12 个条件（金额门槛、数量门槛、用户组、配送方式、渠道、时段、星期几、每月日期、首单、支付方式、商品组合、用户分群→改为静态分群）
- 动作插件基类 + 内置 12 个动作（固定减免、百分比折扣、固定价、阶梯价、阶梯金额、免运费、积分抵扣、余额抵扣、预售定金膨胀、折扣限额、分期免息、随机立减）
- 范围插件基类 + 内置 5 个范围（全部商品、指定 SKU、指定品类、指定标签、排除 SKU）
- 配置驱动插件（`ConfigDrivenCondition` / `ConfigDrivenAction`）—— JSON 配置即可扩展简单逻辑，无需写代码
- 自定义插件注册 API：`engine.register_condition(MyCondition())`

**关键剔除**:
- `ProductPoolScope`（动态商品池）→ 商品池需要数据库和定时同步，过重，开源改为静态 SKU 列表
- `UserSegmentCondition`（动态用户分群）→ 分群需要实时计算用户标签，过重，开源改为静态用户组匹配
- 从数据库主数据表动态加载插件类型 → 开源改为代码注册或 JSON 配置注册

### 2.5 价格保护（精简版）✅

**用户疑问**: 价格保护是不是应该引导到 MyPromotion？

**结论**: **保留核心检查算法，剔除运营配置层**。价格保护的数学公式是引擎能力，不是 SaaS 专属。

**开源形态**（纯函数，输入输出，无数据库）:

```python
from promotion_engine.price_protection import check_price_protection

conflicts = check_price_protection(
    items=[
        {"sku": "SKU001", "sale_price": Decimal("89.00"), "cost_price": Decimal("90.00")},
    ],
    rules={
        "cost_protection": {"min_gross_margin": Decimal("5.00"), "action": "block"},
        "member_protection": {"threshold": Decimal("95.00"), "action": "warn"},
    }
)
# 返回冲突列表，供调用方决定是拦截还是警告
```

**保留算法**:
- 成本价保护：`最低售价 = 成本价 × (1 + 最低毛利率%)`
- 会员价保护：`最低售价 = 会员价 × 阈值%`
- 冲突返回结构（类型、SKU、售价、保护线、动作建议）

**剔除**:
- Django Admin 配置界面
- 多级别会员价动态查询（需要接入用户画像服务）
- `PriceProtectionRule` ORM 模型及数据库表
- 租户默认规则自动回退逻辑

---

### 2.6 售后计算（分摊策略核心）✅

**用户疑问**: 售后计算是不是应该引导到 MyPromotion？

**结论**: **保留三种退款分摊算法的数学实现，剔除 trace 持久化和执行层**。售后分摊是促销计算结果的延续，属于引擎数学能力。

**开源形态**（纯函数，基于计算结果做后处理）:

```python
from promotion_engine.refund import calculate_item_discounts, RefundStrategy

# 正向计算后，得到订单商品和总优惠
item_discounts = calculate_item_discounts(
    order_items=cart_items,
    total_discount=Decimal("50.00"),
    strategy=RefundStrategy.PROPORTIONAL,  # 或 KEEP_DISCOUNT / FULL_REFUND_DISCOUNT
)
# 返回每个 SKU 的 original_price / allocated_discount / payable
```

**保留算法**:
| 策略 | 开源实现 | 说明 |
|-----|---------|------|
| `proportional` | `_calculate_proportional()` | 按比例分摊优惠到每个 SKU，退款时按数量比例收回 |
| `keep_discount` | 优惠不退还 | 退款时实退金额 = 原价，商家承担优惠成本 |
| `full_refund_discount` | 优惠全退 | 退该商品时，该规则给该商品的全部优惠都收回 |

**保留能力**:
- 商品级优惠分摊明细（`item_discounts`）
- 退款金额按数量比例计算
- 截断到剩余可退金额 + 明细重分配（避免精度矛盾）

**剔除**:
- `PromotionTrace` 模型及 MySQL/Redis 存储
- `RefundExecution` 模型及幂等写入
- 乐观锁并发控制
- 冷存归档/回读逻辑
- 券退回/回收策略（依赖外部券系统）
- fallback 触发审计日志

### 2.7 规则配置（静态化）✅

**保留理由**: 引擎需要输入规则，规则配置格式是引擎接口的一部分。

**开源形态**（dataclass + JSON Schema，无数据库）:

```python
from promotion_engine import Rule, RuleCondition, RuleAction, RuleScope

rule = Rule(
    promotion_code="SUMMER2026",
    strategy_type="full_reduction",
    priority=100,
    conditions=[
        RuleCondition(condition_type="min_order_amount", config={"amount": 300})
    ],
    actions=[
        RuleAction(action_type="fixed_amount_reduction", config={"amount": 50}, max_discount=50)
    ],
    scopes=[
        RuleScope(scope_type="all_items", config={})
    ],
    stack_config={"mutex_with": ["WINTER2026"]},
)
```

**保留能力**:
- `Rule` / `RuleCondition` / `RuleAction` / `RuleScope` dataclass
- `stack_config` 叠加配置（`force_stackable`, `stackable_with`, `mutex_with`, `ignore_mutex_groups`）
- 策略-条件-动作-范围的合法性校验（如满减只能用固定减免）
- JSON Schema 配置验证

**剔除**:
- `TenantAwareModel` 多租户基类
- 版本控制（`version`, `is_latest`, `parent_rule`）
- Django Admin 表单/校验
- 审计字段（`created_by`, `updated_by`）
- 标签系统（`PromotionTag` ManyToMany）

---

### 2.8 静态计算（无外部依赖）✅

**保留理由**: 这是开源库的根本承诺——`pip install` 就能跑，不需要 MySQL/Redis/Celery。

**开源形态**:
- 纯标准库 + `decimal` + `dataclasses` + `typing`
- 零框架依赖（无 Django/Flask/FastAPI 导入）
- 计算不访问网络、不查数据库、不发消息队列
- 所有外部数据（用户信息、商品属性、库存状态）通过 `CalculationContext` 显式传入

## 三、明确剔除的功能（留在闭源 SaaS）

### 3.1 用户动态分群 ❌

**原因**: 动态分群需要实时计算用户画像、行为标签、规则引擎查询数据库，属于运营平台能力。

**闭源 SaaS 保留**:
- `apps/user_segment/` 完整模块
- 基于用户行为数据的动态分群计算
- 分群成员实时刷新（Celery 定时任务）

**开源替代**:
- 条件插件保留 `user_group` 静态字段匹配（调用方提前算好用户组，传入 context）
- 如需动态分群，开源 README 引导至 MyPromotion SaaS

### 3.2 动态商品池 ❌

**原因**: 商品池需要维护大量 SKU 数据、定时同步、版本管理，属于运营平台能力。

**闭源 SaaS 保留**:
- `apps/product_pool/` 完整模块
- `ProductPoolRule` / `ProductPoolMember` ORM 模型
- 商品池定时同步任务

**开源替代**:
- 范围插件保留 `specific_items`（静态 SKU 列表）和 `category_items`（静态品类 ID）
- 如需动态商品池，开源 README 引导至 MyPromotion SaaS

### 3.3 Trace / 退款执行 / 审计日志 ❌

**原因**: 需要 MySQL + Redis + Celery + 归档存储，是完整的 SaaS 基础设施。

**闭源 SaaS 保留**:
- `apps/trace/` 完整模块
- `PromotionTrace` / `RefundExecution` / `RefundFallbackLog` 模型
- 快照异步保存、Redis 缓存、冷存归档
- 退款执行的乐观锁并发控制

**开源替代**:
- 保留 `calculate_item_discounts()` 纯函数（见 2.6）
- 调用方自行决定如何存储计算快照（内存、本地文件、或接入 SaaS）

### 3.4 多租户系统 ❌

**原因**: 租户隔离是 SaaS 商业模式的核心，与纯计算引擎无关。

**闭源 SaaS 保留**:
- `apps/tenants/` 完整模块
- `Tenant` 模型、租户中间件、上下文管理
- 数据隔离、权限控制、申请审批流程

**开源替代**:
- 引擎完全不感知租户，一条规则就是一个 `Rule` dataclass
- 调用方自行管理多租户（如需）

### 3.5 Admin 后台 / 生产级 Web UI ❌

**原因**: 开源核心是库，不是应用。完整的规则管理后台属于 SaaS 产品竞争力。

**闭源 SaaS 保留**:
- Django Admin 全部定制界面
- 测试中心、Dashboard、日志查看器
- 租户隔离的权限系统

**开源替代——Demo 演示页面 ✅**:
- 提供一个 `demo/` 目录，包含**交互式演示页面**（FastAPI + 纯 HTML/JS）
- 定位为**"交互式文档"**，不是**"管理系统"**
- 刷新即清空，不持久化任何数据
- 左侧输入购物车 + 规则，右侧实时展示计算结果和跳过原因

**Demo 页面 vs Admin 后台的区别**:

| 维度 | Demo 页面（开源） | Admin 后台（闭源 SaaS） |
|-----|-----------------|---------------------|
| 定位 | 引擎能力展示 + 开发者调试 | 生产环境运营管理 |
| 持久化 | ❌ 无，纯内存/前端 | ✅ 数据库保存规则 |
| 使用者 | 开发者、技术决策者 | 运营人员、商家 |
| 与 SaaS 关系 | ✅ 互补（试用后想买 SaaS） | ❌ 本身就是 SaaS |
| 维护成本 | 低（一个 HTML） | 高（权限、多租户、数据迁移） |

**pip install 后如何访问 Demo（三种路径）**:

```bash
# 方式一：本地启动（推荐首次体验）
pip install mypromotion-engine-core[demo]
mypromotion-engine-demo
# → 自动打开浏览器 http://127.0.0.1:8000/demo/

# 方式二：Python 模块启动（无需额外命令注册）
python -m promotion_engine.demo --port 8080

# 方式三：直接访问在线 Demo（零安装）
# README 提供部署在 Render/Railway 的公共 Demo 链接
# https://promotion-engine-demo.onrender.com
```

`[demo]` 可选依赖仅安装 `fastapi` + `uvicorn`，不影响核心包的零依赖承诺。

### 3.6 Celery 异步任务 / Webhook / 审计 ❌

**原因**: 任务调度、消息推送、审计追踪属于业务编排层。

**闭源 SaaS 保留**:
- 所有 `tasks.py`（Celery 任务）
- Webhook 推送、邮件通知
- 规则审计日志（`RuleAuditLog`）
- API 调用日志（`APICallLog`）

## 六、快速决策 Checklist

当不确定某个功能是否放入开源核心时，逐条检查：

- [ ] 该功能是否纯数学/逻辑计算，不需要数据库/Redis/网络？
- [ ] 该功能是否可以用 dataclass 替代 Django Model 实现？
- [ ] 该功能是否 `pip install` 后无需配置文件即可运行？
- [ ] 该功能是否是促销计算的"不可分割"部分（如互斥、精度、顺序）？
- [ ] 该功能去掉后，引擎是否还能被称为"生产级促销引擎"？
